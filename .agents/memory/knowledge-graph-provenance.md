---
name: Atomic knowledge graph provenance
description: How distilled atomic-knowledge nodes attach to the Living Memory graph, and the intelligence-layer invariants that keep re-processing idempotent and convergent.
---

Atomic knowledge (concept/tool/hazard/etc.) lives in the same `knowledge_nodes`/`knowledge_edges` tables as the core/topic/competency/video scaffold, distinguished by node `kind`. A concept node's canonical id is `k:<category>:<normalized-title>`, so the same concept from any video collapses onto one shared node.

**Provenance is edge-owned, not node-owned.** A videoŌåÆknowledge edge has `kind='knowledge'` and carries `{timestamps, confidence, trade, competencyCode}` in edge `meta`. A concept "belongs to" a video only through this edge; the node itself is shared many-to-many. Every node-level aggregate (confidence, source list, hub-edge weights) is *derived* from these edges, never incremented in place.

**Duplicate detection / merge.** Before writing, each distilled item resolves to a canonical node: (1) exact normalized-label id already exists ŌåÆ reuse it; (2) else `match_knowledge_nodes` RPC finds a same-category node with cosine Ōēź `KNOWLEDGE_MATCH_THRESHOLD` (0.85) ŌåÆ merge onto it; (3) else mint new. Items resolving to the same canonical id merge (union timestamps, max confidence, most-complete description, first competency). A node's `embedding` is only written when the item *owns* the canonical id (exact/new) ŌĆö never when it merely merged onto another node, or the canonical embedding would drift.

**Convergence via recompute.** `recomputeKnowledgeAggregates(ids)` rebuilds, from provenance edges only: node `confidence` = noisy-OR (`1-ŌłÅ(1-cßĄó)`) over distinct source videos; node `meta` = `{sourceVideoIds, sourceCount, timestamps, sources[]}`; each conceptŌåÆtopic / conceptŌåÆcompetency hub-edge `weight` = distinct corroborating source count, deleting any hub edge that drops to weight 0. Because it derives (not increments), replays never double-count. It runs at the end of `syncVideoKnowledge` (canonical ids Ōł¬ prior targets), `removeVideoGraph` (targets captured *before* the cascade delete), and `rebuildGraph` (ALL atomic nodes ŌĆö rebuild is deterministic/free and does not re-distill, so this self-heals a pre-intelligence-layer DB).

**Knowledge Provenance ledger (why each concept exists).** The provenance-engine fields all live in node `meta` (no new SQL columns) and ŌĆö except human verification ŌĆö are DERIVED from edges every recompute, so they stay idempotent under replay/rebuild:
- Extraction provenance: each provenance edge carries `{model, extractedAt}` (the distilling model + ISO date; `syncVideoKnowledge(videoId, items, {model, extractedAt})`, defaulted by the distillation pipeline to `MODELS.analysis`). Recompute derives per-source `{model, extractedAt}` into `sources[]`, plus node-level `models[]` (distinct), `firstExtractedAt`/`lastExtractedAt`. **Pre-feature edges lack these ŌåÆ keep them null, never fabricate a default**, so `models[]`/dates stay honest.
- `confidenceHistory[]` `{confidence, sourceCount, at}` ŌĆö append **only when derived confidence actually changes** (compare to last entry). Idempotent replays and full rebuilds append nothing.
- `mergedFrom[]` `{id, label, category, at}` ŌĆö recorded in `resolveCanonicalItems` when `item.id !== canonicalId` (a differently-worded concept collapsing on). An exact normalized-label match is identity, NOT a merge, and records nothing. First-seen `at` wins; dedupe by id.
- `rejectedEvidence[]` `{videoId, at, reason:'no-longer-extracted'}` ŌĆö written to SURVIVING dropped nodes (priorTargets minus new canonicalIds) AFTER prune / BEFORE recompute, deduped by videoId. Recompute reconciles an entry away once that video corroborates the concept again (drop-then-reteach clears it).
- `verificationHistory[]` `{from, to, at}` ŌĆö the ONE meta field not edge-derived; written by `setNodeVerification` only on a genuine status change. **No reviewer identity by design** (that is a separate signed-in-reviewer feature).
- All history arrays are tail-capped via `capHistory` (`HISTORY_CAP=50`) so churn can't bloat meta.
- **Meta-preservation trap:** the `syncVideoKnowledge` node upsert must spread `{...prevMeta, category, mergedFrom}` (and recompute must spread `{...prevMeta, ...derived}`). Overwriting meta with just `{category}` wipes the whole audit trail on every re-processing of ANY corroborating video ŌĆö this was a caught bug.

**Invariants (violating any breaks idempotent re-processing):**
- The video-node structural edge reconciliation must NOT delete `kind='knowledge'` edges (it excludes them with `.neq("kind","knowledge")`). Those edges are owned by the distillation engine; a plain metadata re-sync that does not re-run distillation must leave them intact.
- `syncVideoKnowledge` reconciles ONLY the current video's `kind='knowledge'` edges (delete source=videoNode & kind=knowledge, then reinsert). It must never touch another video's provenance edges.
- KnowledgeŌåÆtopic / knowledgeŌåÆcompetency edges are additive upserts on ingest (a growing many-to-many web); their *weights* are set authoritatively by the recompute, and only the recompute deletes them (when corroboration hits 0).
- Node merge is first-writer-wins for label/trade; description is most-complete (longest non-empty); confidence and hub weights are recompute-derived (do not hand-set them as the source of truth ŌĆö the interim value in the upsert is immediately overwritten); a human `verification_status` of `verified`/`rejected` is preserved (new nodes default `unverified`).
- `pruneOrphanKnowledge` deletes any atomic node with zero incoming `kind='knowledge'` edges. Call it before the recompute in `syncVideoKnowledge`/`removeVideoGraph`/`rebuildGraph` ŌĆö but NEVER mid-sync (the deleteŌåÆreinsert window legitimately leaves a node edgeless). Recompute skips already-pruned ids (it filters to `k:`-prefixed ids that still exist).

**Why:** the API uses the Supabase service-role key and has no auth, so all graph mutation is pipeline-internal (invoked from `runAnalysis` in videos.ts via best-effort `distillGraphSafe`); there is no public graph-mutation route. A distillation failure must never downgrade a successfully transcribed/analyzed video.
